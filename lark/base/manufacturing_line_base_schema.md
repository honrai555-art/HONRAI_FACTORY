# 製造ライン Base 設計

## 概要

Lark Base は HONRAI_FACTORY の製造ラインを見える化するための DB 設計です。  
Google スプレッドシートや DB に本体データを置き、Lark Base では案件・タスク・担当者・成果物・エラー・週次レポートの状態を管理します。

---

## 1. 案件管理

### 目的
案件の全体進捗を管理し、種類、担当者、期限、成果物、次のアクションを一元化します。

### カラム
- `案件ID`：案件を一意に識別するID
- `案件名`：案件のタイトル
- `種類`：診断、キャリアカード、漫画、ゲーム、メタバース、財務、運営、エラー対応、週次報告 など
- `担当者`：主担当者名
- `状態`：案件の進行状態
- `締切`：完了予定日
- `成果物URL`：成果物の保管場所URL
- `エラー内容`：発生中のエラーや問題点
- `次のアクション`：次に必要な対応
- `更新日`：最終更新日

### 入力例
- `案件ID`: PRJ-001
- `案件名`: 診断回答が入りました
- `種類`: 診断
- `担当者`: Cさん
- `状態`: 作業中
- `締切`: 2026-05-28
- `成果物URL`: https://docs.google.com/...
- `エラー内容`: なし
- `次のアクション`: AI分析開始
- `更新日`: 2026-05-26

### ステータス候補
- 未着手
- 作業中
- 確認待ち
- 修正中
- 完了
- 停止中
- エラー

### 誰が更新するか
- 担当者本人
- 管理者（進捗を確認して更新）
- QA 担当（確認待ち／修正中を反映）

- `締切`：期限
- `成果物URL`：関連成果物のURL
- `エラー内容`：発生した課題や障害
- `次のアクション`：次の手順
- `更新日`：最終更新日

- `案件ID`: PRJ-003
- `タスク名`: 漫画セリフ確認
- `種類`: 漫画
- `担当者`: Aさん
- `状態`: 確認待ち
- `締切`: 2026-05-27
- `成果物URL`: https://drive.google.com/...
- `エラー内容`: なし
- `次のアクション`: セリフ修正
- `更新日`: 2026-05-26

### ステータス候補
- 未着手
- 作業中
- 確認待ち
- 修正中
- 完了
- 停止中
- エラー

### 誰が更新するか
- タスク担当者
- リーダー
- レビュー担当

## 3. 担当者管理

### 目的
アルバイトやメンバーの役割、現在の担当状況、連絡先を管理します。

### カラム
- `担当者ID`：担当者識別子
- `担当者名`：氏名またはコード名
- `役割`：業務分担（漫画確認、素材整理、PDF 確認）
- `現在の担当案件`：主な案件名
- `稼働状態`：作業可能 / 作業中 / 休止
- `連絡先`：連絡手段（Lark / Discord / メール）
- `更新日`：最終更新日

### 入力例
- `担当者ID`: A01
- `担当者名`: Aさん
- `役割`: 漫画セリフ確認
- `現在の担当案件`: PRJ-003
- `稼働状態`: 作業中
- `連絡先`: Lark
- `更新日`: 2026-05-26

### ステータス候補
- 作業可能
- 作業中
- 休止
- 応答待ち

### 誰が更新するか
- 担当者自身
- リーダー
- 運用管理者

## 4. 成果物管理

### 目的
生成済み成果物の種類、保存場所、確認状態、関連案件を管理します。

### カラム
- `成果物ID`：成果物識別子
- `案件ID`：関連案件ID
- `成果物名`：ファイル名や成果物名
- `種類`：PDF、漫画、画像、素材、ビルドなど
- `保存場所`：URL またはフォルダパス
- `確認状況`：未確認 / 確認済み / 修正要
- `ファイル形式`：PNG、PDF、ZIP など
- `作成日`：生成日
- `更新日`：最終更新日

- `案件ID`: PRJ-002

- `種類`: PDF
- `保存場所`: https://drive.google.com/...
- `確認状況`: 確認済み
- `ファイル形式`: PDF
- `作成日`: 2026-05-26
- `更新日`: 2026-05-26

### ステータス候補
- 未確認
- 確認中
- 確認済み
- 修正要
- 完了

### 誰が更新するか
- 成果物作成担当
- 確認担当
- QA 担当

## 5. エラー管理

### 目的
システムや作業上のエラーを記録し、再発防止と対応状況の追跡を行います。

### カラム
- `エラーID`：エラー識別子
- `案件ID`：関連案件ID
- `発生日時`：エラー発生日時
- `エラー種別`：GAS、ComfyUI、Discord Bot、その他
- `内容`：エラーメッセージ・状況
- `影響範囲`：影響を受ける工程
- `対応状況`：未対応 / 対応中 / 対応済み
- `担当者`：対応担当者
- `対応内容`：対応の詳細
- `更新日`：最終更新日

### 入力例
- `エラーID`: ERR-001
- `案件ID`: PRJ-001
- `発生日時`: 2026-05-26 15:00
- `エラー種別`: GAS
- `内容`: スプレッドシート読み込み失敗
- `影響範囲`: 診断データ取り込み
- `対応状況`: 対応中
- `担当者`: 運営
- `対応内容`: アクセス権確認中
- `更新日`: 2026-05-26

### ステータス候補
- 未対応
- 対応中
- 対応済み
- 再発監視

### 誰が更新するか
- 障害対応担当
- 運用管理者
- 技術担当

## 6. 週次レポート

### 目的
週次で製造ラインの状況をまとめ、関係者に報告するための記録を保持します。

### カラム
- `報告週`：週次レポート対象の週
- `生成件数`：週内で完了した生成件数
- `エラー件数`：週内のエラー発生件数
- `保留件数`：週内の保留・確認待ち件数
- `作業時間`：週次の工数見積もり
- `重要メモ`：特記事項
- `次週対応`：翌週の優先課題
- `作成者`：報告者
- `更新日`：最終更新日

### 入力例
- `報告週`: 2026-W21
- `生成件数`: 18 件
- `エラー件数`: 2 件
- `保留件数`: 3 件
- `作業時間`: 48h
- `重要メモ`: ComfyUI のメモリ不足発生
- `次週対応`: モデル再配置 / データ整理
- `作成者`: 運営
- `更新日`: 2026-05-26

### ステータス候補
- 作成中
- 完成
- 共有済み

### 誰が更新するか
- 運営管理者
- 週次報告担当

---

## 運用メモ

- Lark Base では各テーブルを最小限にして、運用中のステータスと次のアクションがすぐに分かるようにします。
- Google スプレッドシートや DB がデータの本体であり、Lark Base は見える化のためのサマリーデータとして扱います。
- 関係者は `更新日` を必ず書き、誰が最後に更新したかを明確にします。
