# Lark Docs 運用ルール

## 目的
Lark Docs を HONRAI_FACTORY のナレッジ共有と運用標準として使うためのルールを定義します。

## 1. ドキュメントカテゴリ

- `運用設計`：Base / Docs / Rules の構造
- `通知テンプレート`：Lark / Discord / 社内連絡の定型文
- `作業手順`：ComfyUI 生成、画像生成、エラー復旧、週次報告
- `品質基準`：成果物チェックリスト、承認ルール
- `更新履歴`：変更点と更新者

## 2. ドキュメントの書き方

- タイトルは簡潔に。例：`ComfyUI 画像生成手順`。
- 目的、対象、手順、備考を明確に分ける。
- 変更時は `更新日` と `更新者` を追記する。
- 重要な手順は箇条書きで記載する。
- 画像やコードブロックは必要最小限にする。

## 3. 更新ルール

- ドキュメント更新は必ず `変更概要` を一行で追記する。
- 大きな変更は `週次報告` で共有する。
- ドキュメント内容が古い場合は `更新待ち` ステータスをつける。
- 仕様変更時は `Lark Base` の該当項目も併せて更新する。

## 4. 参照の統一

- `lark/base` はデータ構造・マスタ定義
- `lark/docs` は設計思想・連携フロー
- `lark/rules` は運用ルール・通知基準
- `README.md` は全体の入口として機能させる

## 5. 標準化ルール

- 用語の定義を統一する。例：`案件`、`タスク`、`成果物`。
- ステータス名称は `status_master.csv` と一致させる。
- URL やパスは可能な限り相対リンクではなく、実際の参照先を示す。
- 英語と日本語を混在させる場合は、補足説明を必ず記載する。

## 6. 保守運用

- 週次で `Docs` 内容を見直し、古い情報を削除または更新する。
- 運用担当は `Docs` の改善点をまとめて `週次レポート` に提出する。
- 重要な運用変更は `Lark 通知` で共有する。

## 7. 例外対応

- 仕様変更が緊急の場合、まず `Lark` で通知を行い、その後 `Docs` を更新する。
- 一時的な運用ルールは `一時ルール` として別途管理し、期限付きで廃止する。
